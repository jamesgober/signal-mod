# signal-mod v1.0.0 - Production Release

**Date:** 2026-05-19
**Compare:** `v1.0.0-rc.1...v1.0.0`

## Headline

The first production-stable release of `signal-mod`. Every public
item re-exported from the crate root is covered by semantic
versioning from this tag forward; the next public-surface change is
either a `1.x.0` minor (additions only) or a `2.0.0` major.

The release rolls up the cumulative pre-1.0 work (`0.2.0` design
lock through `0.9.1` MSRV patch, plus the `1.0.0-rc.1` API freeze
tag from earlier in the same session) and adds a substantial round
of production polish:

1. **Crate rename `mod-signal` -> `signal-mod`** to match the
   repository name at `github.com/jamesgober/signal-mod`. The
   library import is now `signal_mod`.
2. **Modular installer back-ends** under `src/install/`. The
   monolithic install body inside `coord.rs` is replaced with three
   feature-gated submodules (`tokio_rt`, `async_std_rt`,
   `ctrlc_sync`) plus a thin dispatch table.
3. **Panic-safe hook execution.** `Coordinator::run_hooks` now wraps
   each hook in `std::panic::catch_unwind`; a panic in one hook no
   longer aborts the remaining sequence.
4. **Expanded test surface.** 20 new edge-case tests, 6 new stress
   tests, 7 runnable examples covering every documented usage
   pattern.
5. **Full documentation rewrite.** `README.md`, `docs/API.md`, and
   `docs/MIGRATION.md` are now production-quality with install
   instructions, the feature matrix, performance numbers, a
   platform-support grid, the testing summary, and per-item API
   reference with multiple code examples per use case.
6. **`Cargo.toml` metadata tuned for crates.io discoverability.**
   Description rewritten, keywords reordered for search intent,
   categories rebuilt from the crates.io taxonomy.

## What changed

### Added - production 1.0 public API

The public surface is now contractual:

- `Signal`, `SignalSet`, `SignalSetIter`
- `ShutdownReason`
- `ShutdownToken`, `ShutdownTrigger`
- `ShutdownHook`, `FnHook`, `hook_from_fn`
- `Coordinator`, `CoordinatorBuilder`, `Statistics`
- `Error`, `Result`
- `VERSION`

Every item carries rustdoc with `# Examples` and (where applicable)
`# Errors`. Offline mirror in `docs/API.md`.

### Added - `src/install/` modular back-ends

```text
src/install/
├── mod.rs              dispatch documentation
├── tokio_rt.rs         tokio::signal + tokio::spawn
├── async_std_rt.rs     signal-hook-async-std on Unix; ctrlc on Win
└── ctrlc_sync.rs       synchronous ctrlc fallback
```

`Coordinator::install_impl` becomes a four-line `cfg`-gated
dispatch. The previous monolithic install body in `coord.rs`
(roughly 200 lines of Unix / Windows macros plus async-std fallback)
is gone.

### Added - panic-safe hook execution

```rust
let result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
    hook_ref.run(reason);
}));
if let Err(_panic) = result {
    // Swallow; the next hook still runs.
}
count += 1;
```

A new `tests/edge_cases.rs::panicking_hook_does_not_abort_remaining`
test guards the contract. The hook is counted as "completed" either
way, because from the coordinator's perspective the hook's
lifecycle ran to a terminal state.

### Added - 7 runnable examples

| File                                                 | Demonstrates                                          |
| ---------------------------------------------------- | ----------------------------------------------------- |
| `examples/graceful_shutdown.rs`                      | Canonical install + wait + run_hooks pattern.         |
| `examples/programmatic_shutdown.rs`                  | Trigger without OS signals.                           |
| `examples/multi_subsystem.rs`                        | Multiple observer tasks fanning out.                  |
| `examples/sync_blocking.rs`                          | Non-async CLI with `ctrlc-fallback`.                  |
| `examples/custom_signal_set.rs`                      | `SignalSet` at compile time and runtime.              |
| `examples/priority_hooks.rs`                         | Hook priority + graceful budget.                      |
| `examples/custom_hook_type.rs`                       | Implementing `ShutdownHook` directly.                 |

### Added - 26 new tests

- `tests/edge_cases.rs` (20 tests): empty signal sets, double
  install, zero / large hook counts, panicking hooks, extreme
  priorities, immediate-return waits, install-without-back-end.
- `tests/stress.rs` (6 tests): 32-thread trigger race, 16-observer
  fan-out, 10 000-clone consistency, 256-cycle trigger/observe
  loop, concurrent `run_hooks` from 8 threads, wait responsiveness
  under contention.

### Added - production documentation

- `README.md` now ships Install, Why, Features, Examples, API at a
  glance, Performance, Platform support, Testing, MSRV, and
  Contributing sections. Pre-1.0 status notices removed; install
  snippets pin `"1.0"`.
- `docs/API.md` is a full per-item reference. Every type, trait,
  and function has a description, a parameter / return table, and
  at least one runnable code example.
- `docs/MIGRATION.md` rewritten as a 1.0 adoption and `1.x` upgrade
  guide. Includes a "From a custom `ctrlc` + `signal-hook` setup"
  recipe and a "From `tokio::signal::ctrl_c()`" recipe.
- `REPS.md` section 4 retitled "1.x.y"; section 8 (Stability)
  rewritten as the production semver contract.

### Changed - crate name `mod-signal` -> `signal-mod`

The crate was renamed to match the repository at
`github.com/jamesgober/signal-mod`. The library import is now
`signal_mod` (snake_case).

| Field                   | Before                   | After                       |
| ----------------------- | ------------------------ | --------------------------- |
| Cargo `name`            | `mod-signal`             | `signal-mod`                |
| Cargo `[lib] name`      | `mod_signal`             | `signal_mod`                |
| `use` statements        | `use mod_signal::*`      | `use signal_mod::*`         |
| `Cargo.toml` repository | `.../mod-signal`         | `.../signal-mod`            |
| docs.rs URL             | `docs.rs/mod-signal`     | `docs.rs/signal-mod`        |

Migration for pre-1.0 consumers: change the `Cargo.toml`
dependency line from `mod-signal = "..."` to `signal-mod = "1.0"`
and run `cargo fix --edition` (or do a global search/replace) to
update `use mod_signal::*` to `use signal_mod::*`.

### Changed - `Cargo.toml` metadata

| Field           | Before                                                                                                                                                                                                          | After                                                                                                                                                                                                                                                                                       |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Description** | "Unified OS signal handling for Rust. SIGTERM / SIGINT / SIGHUP / SIGPIPE and Windows equivalents through one API. Graceful shutdown orchestration with priority ordering, timeout enforcement, and hook chaining. ..." | "Cross-platform OS signal handling and graceful-shutdown orchestration for Rust. One API for SIGTERM, SIGINT, SIGHUP, SIGQUIT, SIGPIPE, SIGUSR1, SIGUSR2 and the Windows console control events, with cloneable observer and initiator handles, priority-ordered shutdown hooks, and optional adapters for the Tokio and async-std runtimes." |
| **Keywords**    | `signal`, `sigterm`, `shutdown`, `ctrlc`, `process`                                                                                                                                                             | `signal`, `sigterm`, `shutdown`, `graceful`, `ctrlc`                                                                                                                                                                                                                                        |
| **Categories**  | `os`, `concurrency`                                                                                                                                                                                              | `os`, `os::unix-apis`, `os::windows-apis`, `asynchronous`, `concurrency`                                                                                                                                                                                                                    |

Rationale:

- `process` was misleading - the crate does not spawn or manage
  child processes; that is downstream daemon-framework territory.
  Replaced with `graceful` to capture the shutdown-orchestration
  intent.
- The two new `os::unix-apis` and `os::windows-apis` categories are
  accurate (the crate has dedicated back-ends for each) and improve
  category-page placement.
- `asynchronous` is added because the default runtime adapter is
  async.

### Changed - REPS, MIGRATION, README aligned to 1.0

Every pre-1.0 status note, "release candidate" reference, "API
freeze coming" wording, and `0.x.y` reference removed from the
shipping docs. Version references all pin to `1.0`.

### Internal

- `coord.rs` shrunk by extracting installer bodies into
  `src/install/`. The remaining file is the state machine, hook
  list, statistics snapshot, and builder.
- Hook execution is now panic-safe via `std::panic::catch_unwind`.
- `forbid(unsafe_code)` retained crate-wide. All `unsafe` is
  delegated to `tokio`, `signal-hook`, `signal-hook-async-std`,
  and `ctrlc`.

## Migration from `1.0.0-rc.1`

If you pinned `signal-mod = "=1.0.0-rc.1"`, change the pin to
`"1.0"` (or `"=1.0.0"` for an exact match). No source-level
migration is required; the public surface is the same, with the
addition of the panic-safety contract on `run_hooks` (any code that
relied on a panicking hook unwinding past the coordinator must
catch its panics inside the hook body now).

## Migration from `mod-signal` (`0.x.y`)

1. Change the dependency line from `mod-signal = "..."` to
   `signal-mod = "1.0"`.
2. Replace every `use mod_signal::` with `use signal_mod::`.
3. If you were pinned to a pre-1.0 release, also verify the
   public-API rename hasn't changed any symbols you depend on -
   the surface is unchanged from `0.9.1` aside from the crate-name
   change itself.

`docs/MIGRATION.md` carries the full list of recipes.

## Stability commitment

`1.0.0` is the production-stable line. Going forward:

- **Patch (`1.x.y`)** - bug fixes, doc improvements, dep bumps in
  semver-compatible ranges, CI fixes. No public surface change.
- **Minor (`1.x.0`)** - pure additions, new opt-in features,
  internal perf. MSRV bumps allowed (the current MSRV is `1.75`).
- **Major (`2.0.0`)** - breaking changes only.

The `Error` enum is `#[non_exhaustive]`, so new variants are a
minor-version change. Downstream `match` arms must include a
wildcard.
