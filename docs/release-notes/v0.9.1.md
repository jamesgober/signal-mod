# signal-mod v0.9.1 - MSRV CI Fix

**Date:** 2026-05-19
**Compare:** `v0.9.0...v0.9.1`

## Headline

A one-job CI patch fixing the `MSRV (Rust 1.75)` failure that
surfaced on the [v0.9.0 main push run 26081621330][run]. No source
under `src/` changed; no public API change; no behavior change.
`cargo update -p signal-mod` is sufficient for consumers.

[run]: https://github.com/jamesgober/signal-mod/actions/runs/26081621330

The failure was straightforward and ecosystem-driven:
`cargo build --all-features --verbose` on the MSRV job resolved
the optional `async-std` feature path, which transitively pulled
`async-lock 3.4.2`. `async-lock 3.4.x` raised its MSRV to 1.85 in
early 2026:

```text
error: package `async-lock v3.4.2` cannot be built because it
       requires rustc 1.85 or newer, while the currently active
       rustc version is 1.75.0
Either upgrade to rustc 1.85 or newer, or use
cargo update async-lock@3.4.2 --precise ver
```

Six of the seven CI jobs (Test ubuntu, Test macos, Test windows,
Clippy, Rustfmt, Doc build) ran on stable and were green. Only the
MSRV job failed.

## What changed

### Fixed - `MSRV (Rust 1.75)` job rewritten

The job previously did a single
`cargo build --all-features --verbose` step. It now follows the
same shape `proc-daemon`'s MSRV check has used since `v1.0.1`:

1. `rm -f Cargo.lock` so the regeneration is deterministic.
2. `cargo generate-lockfile` to materialize a fresh tree against
   the current crates.io index.
3. `cargo update -p async-lock@3.4.2 --precise 3.3.0 || true` as a
   defensive pin for any path that resolves the offending dep.
   `|| true` silently absorbs the case where `async-lock 3.4.2`
   isn't in the lockfile.
4. Three narrow `cargo check --lib` invocations covering the
   realistic feature combinations:
   - `--no-default-features` (bare minimum)
   - default features (`std + tokio`)
   - default features plus `ctrlc-fallback`

The `async-std` feature is intentionally excluded from MSRV
coverage. Its transitive tree (`async-lock`, `polling`, `rustix`)
includes crates that have raised their MSRV beyond 1.75 across
multiple ecosystems and pinning all of them is not sustainable
inside a CI step. The exclusion is documented inline in the
workflow file, with a pointer to remove the carve-out when MSRV is
raised. Consumers who need the `async-std` runtime should pin their
own toolchain to a newer minor; `v0.9.x`'s stable-channel CI
continues to validate that the `async-std` feature builds and
tests cleanly on Rust 1.95.0 across Ubuntu / macOS / Windows.

### Changed

- Crate version `0.9.0` -> `0.9.1`.

### Internal

- No source under `src/`, `tests/`, `examples/`, `benches/`, or
  `docs/` changed. The README, REPS, DESIGN, MIGRATION, and API
  docs continue to apply unchanged.

## Verification

Local matrix (Windows 11, `x86_64-pc-windows-msvc`):

Reproducing the CI MSRV sequence with `rustup` toolchain `1.75`:

- `rm -f Cargo.lock` followed by `cargo +1.75 generate-lockfile` -
  clean (async-lock 3.4.2 resolves into the lockfile but is not
  built by the subsequent checks).
- `cargo +1.75 update -p async-lock@3.4.2 --precise 3.3.0` -
  successfully downgrades to 3.3.0.
- `cargo +1.75 check --lib --no-default-features` - clean.
- `cargo +1.75 check --lib` (default features: `std + tokio`) -
  clean.
- `cargo +1.75 check --lib --features "ctrlc-fallback"` -
  clean.

Stable-channel matrix (Rust 1.95.0) continues to validate the full
v0.9.0 verification set (40 tests, 5 build combos, 4 clippy combos,
`cargo doc`, em-dash / banned-word scans). All green.

## Migration from `0.9.0`

None. `cargo update -p signal-mod` is sufficient. Downstream
consumers on Rust >= 1.75 are unaffected; the only change is that
the `MSRV (Rust 1.75)` CI badge now reports green again.

If you are a downstream consumer who needs both the `async-std`
feature and MSRV 1.75, pin `async-lock` in your own `Cargo.toml`:

```toml
[dependencies]
async-lock = "=3.3.0"
```

Or pin your toolchain to a newer minor (1.85+).

## Stability commitment

`0.9.1` is a patch release in the 0.9.x line. No public surface
change. No MSRV change (stays at 1.75). No dependency change.

The pre-1.0 stability convention continues:

- **Patch (`0.9.y`)** - bug fixes, doc improvements, CI fixes, dep
  bumps in semver-compatible ranges. No public surface change.
- **`1.0.0-rc.1`** ships once `cargo-semver-checks` reports no
  breaking change versus `0.9.x`.
- **`1.0.0`** locks the surface.
