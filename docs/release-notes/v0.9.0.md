# signal-mod v0.9.0 - Pre-1.0 Stabilization

**Date:** 2026-05-19
**Compare:** `v0.4.0...v0.9.0`

## Headline

The release that opens the pre-1.0 stabilization window. No public
API change versus `0.4.0`. The bump exists to mark the boundary
between active surface evolution (`0.2.0` -> `0.4.0`) and the
freeze prep that culminates in `1.0.0-rc.1`.

Three additions land:

1. **Property tests.** Ten algebraic properties exercised at 256
   cases each, covering `SignalSet`'s set-theoretic identities, the
   first-trigger-then-redundant invariant on `ShutdownTrigger`,
   hook-completion totals, Unix signal-number uniqueness, and the
   platform-availability consistency between `is_unix_only` and
   `available_on_current_platform`.
2. **Migration guide.** `docs/MIGRATION.md` documents the upgrade
   path between every shipped version, the forward-looking
   `1.0.0-rc.1` -> `1.0.0` plan, and how `signal-mod` coexists
   with `ctrlc`, `signal-hook`, and `proc-daemon` while the
   ecosystem catches up.
3. **README rewritten.** The scaffold copy ("repository is published
   primarily to reserve the crate name...") is gone. The README now
   reflects the actual status (`ACTIVE - pre-1.0 stabilization,
   current 0.9.x`), ships the canonical Quick Start, and lists the
   Cargo feature matrix inline.

`cargo update -p signal-mod` is sufficient for consumers. No source
under `src/` changed in this release.

## What changed

### Added - `tests/property_tests.rs`

Ten properties driven by `proptest 1.x`:

- `with(s).contains(s)` is always true.
- `without(s).contains(s)` is always false.
- `with(s).without(s) == without(s)`.
- `without(s).with(s).contains(s)`.
- `len()` matches `iter().count()`.
- `iter()` is in canonical [`Signal::ALL`] order.
- First `trigger` returns `true`; every subsequent trigger returns
  `false`; the first reason is preserved.
- `run_hooks` returns `hooks.len()` when the budget is large.
- `Signal::unix_number` is unique across distinct variants.
- `Signal::is_unix_only` is consistent with
  `available_on_current_platform` on Windows.

The proptest strategies are local helpers
(`any_signal`, `any_signal_set`, `any_reason`) so the tests double
as a small reference for how to fuzz the surface in a downstream
project.

### Added - `docs/MIGRATION.md`

Six sections:

- Migration from `0.2.0` or earlier (design-only releases).
- `0.3.0` -> `0.4.0` (no source change required).
- `0.4.0` -> `0.9.0` (no source change required; new bundled
  property tests can replace any custom ones you wrote).
- Forward-looking `0.9.x` -> `1.0.0` plan including the
  `cargo-semver-checks` gate, the 14-day RC soak window, and a
  list of items that may still change before `1.0.0-rc.1`.
- Coexisting with `ctrlc` and `signal-hook` (don't double-install
  signal handlers; the cleanest path is to retire the direct
  dependency).
- Coexisting with `proc-daemon` (`proc-daemon` v2 will consume
  `signal-mod` internally; until then, pick one to own signal
  installation per process).

### Added - `proptest 1` dev-dependency

Pulled in transitively: `rand 0.9`, `rand_chacha 0.9`, `rand_xorshift 0.4`,
`regex 1.12`, `regex-automata 0.4`, `rusty-fork 0.3`. None of these
appear in downstream consumer builds; `proptest` is dev-only.

### Changed - README

Replaces the v0.1.0 scaffold copy with content that matches the
actual crate state:

- Status line reads `ACTIVE - pre-1.0 stabilization. Current
  release: 0.9.x. API freeze begins at 1.0.0-rc.1.`
- The "What it does" section lists the four pillars (one API,
  graceful orchestration, priority hooks, runtime-agnostic
  substrate).
- A working Quick Start example occupies its own section, with
  links to the runnable `examples/graceful_shutdown.rs`, the
  offline `docs/API.md`, and the new `docs/MIGRATION.md`.
- A Cargo feature table is inline rather than buried.
- MSRV `1.75` and the bump policy are stated in their own section.

### Changed - `examples/graceful_shutdown.rs`

Imports of `std::time::Duration` and the `signal_mod::*` items
are now gated on `feature = "tokio"`. Without the gate, building
the example under `--no-default-features` produced unused-import
errors at clippy's `-D warnings` setting. The gate is purely
mechanical; rendered output is unchanged.

### Changed - `Cargo.toml`

- Crate version `0.4.0` -> `0.9.0`.
- `proptest = "1"` added under `[dev-dependencies]`.

## Migration from `0.4.0`

`cargo update -p signal-mod` is sufficient. There is no source-level
migration. See `docs/MIGRATION.md` for the full upgrade history and
the forward-looking `1.0.0` plan.

## Stability commitment

`0.9.0` is the entry point to the pre-1.0 stabilization window.
The contract going forward:

- **Patch (`0.9.y`)** - bug fixes, doc improvements, dep bumps in
  semver-compatible ranges. No public surface change.
- **`1.0.0-rc.1`** ships once `cargo-semver-checks` reports no
  breaking change versus `0.9.x`.
- A 14-day RC soak period; bugfixes only.
- **`1.0.0`** locks the surface. Every item exported from the crate
  root is covered by semver from then on.

Items that may still evolve between `0.9.0` and `1.0.0-rc.1` are
enumerated in `docs/MIGRATION.md`.
