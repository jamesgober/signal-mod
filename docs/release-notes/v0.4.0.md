# mod-signal v0.4.0 - Polish & Benchmarking

**Date:** 2026-05-19
**Compare:** `v0.3.0...v0.4.0`

## Headline

A benchmarking-and-docs release that closes the `0.4.0` milestone in
`.dev/ROADMAP.md`: realistic micro-benchmarks for every hot path, a
new Performance section in `docs/API.md` with measured numbers, and
the criterion harness wiring needed to make `cargo bench` work. No
public API change. No behavior change.

`cargo update -p mod-signal` is sufficient for consumers; source
compiles unchanged from `0.3.0`.

## What changed

### Added - `benches/shutdown_bench.rs`

A criterion-based bench suite covering the seven measured cases the
new Performance section reports on:

```text
token::is_initiated (uninitiated)              364 ps
trigger::first (transition)                    123 ns
trigger::redundant (already initiated)         4.7 ns
token::clone                                   7.2 ns
run_hooks/hooks=1                              60 ns
run_hooks/hooks=4                              156 ns
run_hooks/hooks=16                             536 ns
run_hooks/hooks=64                             2.08 us
SignalSet::iter (all 7)                        1.5 ns
wait_blocking_timeout (1us, not initiated)     15.4 ms*
```

\* The 15.4 ms figure for `wait_blocking_timeout` at 1 microsecond
reflects Windows timer granularity (`Sleep` / condvar minimum wait
of approximately one scheduler tick), not the primitive itself.
Realistic budgets behave as expected. This is called out in the new
Performance section.

Each bench targets a specific question:

- **`token::is_initiated`**: how fast is the observer fast-path?
  Result: one relaxed atomic load (~one `mov`), sub-nanosecond.
- **`trigger::first` vs `trigger::redundant`**: separates the cost
  of the actual state transition (CAS + mutex + condvar notify +
  broadcast send: ~123 ns) from the no-op redundant path (failing
  CAS only: ~5 ns).
- **`token::clone`**: how cheap is handing out observer handles?
  Result: `Arc::clone` at ~7 ns. Subsystems can hold one each
  without worry.
- **`run_hooks` at 1/4/16/64**: linear scaling characterization.
  Sort overhead is `O(n log n)` but dominated by per-hook dispatch
  at all realistic sizes.
- **`SignalSet::iter`**: 1.5 ns for all seven variants confirms the
  bit-iteration is fully inlined.

### Added - Performance section in `docs/API.md`

A new Performance section under the public-API tables. Includes the
measurement table above, a note on each measured operation's
implementation, and the Windows-timer caveat for tiny
`wait_blocking_timeout` budgets.

### Added - Cargo wiring

- `criterion = { version = "0.5", default-features = false }` as a
  dev-dependency.
- `[[bench]] name = "shutdown_bench" harness = false` so criterion
  drives the bench.
- `[profile.bench] debug = true` to keep useful symbols in flamegraphs
  if someone profiles.

### Changed

- Crate version `0.3.0` -> `0.4.0`.

## Migration from `0.3.0`

No source-level migration. `cargo update -p mod-signal` is
sufficient. The new bench target is a dev-only artifact; downstream
consumers do not pay for `criterion` in their builds.

If you want to reproduce the Performance section on your own
hardware:

```bash
cargo bench --bench shutdown_bench
```

The full criterion HTML report lands in
`target/criterion/report/index.html`.

## Stability commitment

`0.4.0` is a minor release in the 0.x line. No public surface
changed; the bump exists to mark the polish-and-benchmarking
milestone. The pre-1.0 stability convention continues:

- **Minor (`0.x.0`)** - public surface may add and rearrange.
- **Patch (`0.x.y`)** - bug fixes only; no surface change.

API freeze begins at `1.0.0-rc.1`.
