# signal-mod v0.2.0 - Design Lock

**Date:** 2026-05-19
**Compare:** `v0.1.0...v0.2.0`

## Headline

A documentation-only release that closes the design-lock milestone
defined in `.dev/ROADMAP.md`. No source under `src/` changed.
`Cargo.toml` version bumped from `0.1.0` to `0.2.0` so the
crates.io index reflects the milestone boundary; `cargo update -p
signal-mod` is sufficient for consumers tracking pre-1.0 line items.

Three deliverables land:

1. **REPS section 4 is no longer a stub.** Every public type, trait,
   and function signature for the 0.x line is specified, including
   the `Signal` enum and its Windows-event mapping, the
   `SignalSet` bit-packed configuration, the `ShutdownToken` /
   `ShutdownTrigger` capability split, the sync `ShutdownHook` trait,
   the `Coordinator` and its builder, the `Statistics` snapshot type,
   and the `Error` enum with its `Result` alias.
2. **`.dev/DESIGN.md` documents the trade-offs.** Twelve sections
   cover the substrate-vs-framework boundary with `proc-daemon`, the
   capability-split rationale, the choice of a sync hook trait over
   `async-trait`, the `parking_lot` dependency, the optional
   `tokio` / `async-std` features, the sorted-vec hook ordering, the
   install-vs-trigger fallibility asymmetry, the Windows console
   event mapping table, the boundary with `signal-hook` and `ctrlc`,
   and the alternatives that were considered and rejected.
3. **Three named consumer profiles.** REPS section 13 lists the
   target use cases the API was validated against: `proc-daemon`
   (in-tree sibling crate, async daemon framework), CLI long-running
   tools (development servers, log tailers, file watchers), and
   embedded supervisor processes (workers whose parent already owns
   signal delivery and only needs the trigger half).

## What changed

### Added - `REPS.md` section 4 (Public API)

The full 0.x public surface is now spec'd. The shape:

- `Signal` enum: `Terminate`, `Interrupt`, `Quit`, `Hangup`, `Pipe`,
  `User1`, `User2`. Const helpers for description, unix signal
  number, Unix-only check, and platform availability.
- `SignalSet`: bit-packed, const-constructible. Named presets
  `empty()`, `all()`, `graceful()` (Term + Int + Hangup), and
  `standard()` (Term + Int + Quit + Hangup). `with` / `without` /
  `contains` / `iter`.
- `ShutdownReason`: `Signal(Signal)`, `Requested`, `Forced`,
  `Timeout`, `Error`. Display impl plus `description` / `is_signal`
  const helpers.
- `ShutdownToken`: observer handle, cloneable. `is_initiated`,
  `reason`, `elapsed`, `wait_blocking`, `wait_blocking_timeout`, plus
  feature-gated async `wait`.
- `ShutdownTrigger`: initiator handle, cloneable. `trigger` returns
  `bool` (did this call perform the transition).
- `ShutdownHook` trait: sync `run(&self, reason)`, named, with a
  default-zero `priority`. Convenience `hook_from_fn` for closures.
- `Coordinator` and `CoordinatorBuilder`: builder pattern;
  `token()`, `trigger()`, `install()`, `run_hooks()`, `statistics()`.
- `Statistics`: snapshot type with `initiated`, `reason`,
  `hooks_registered`, `hooks_completed`, `elapsed`.
- `Error` non-exhaustive enum: `AlreadyInstalled`,
  `SignalRegistration { signal, source }`, `InvalidState`, `Timeout`,
  `NoRuntime`.

### Added - `.dev/DESIGN.md`

Internal design-trade-off document. Twelve sections covering the
substrate role, the two-handle split, hook-trait shape, choice of
`parking_lot`, optional runtimes, hook ordering, install
fallibility, Windows mapping, library boundaries, and rejected
alternatives.

### Added - REPS section 13 (Real consumers)

Three concrete consumer profiles that informed the API shape. Each
profile is one paragraph: what they want, why the current API gives
it to them, what they would otherwise have to wire up themselves.

### Changed - REPS sections 3, 9

- Section 3 (Scope) now lists the four pillars: `Signal` enum,
  `Coordinator`, observer / initiator handles, runtime adapters.
- Section 9 (Dependency policy) enumerates the current dependency
  set (`parking_lot` mandatory; `tokio`, `async-std`,
  `signal-hook-async-std`, `signal-hook`, `futures`, `ctrlc`, `libc`
  optional) with cross-references to `DESIGN.md`.

### Changed - `Cargo.toml`

Version bumped from `0.1.0` to `0.2.0`. No other change.

- `cargo build` (default features) - clean.

- `cargo clippy --all-targets --all-features -- -D warnings` -

No tests added in this release; the public API is documented but not

## Migration from `0.1.0`

No source-level migration. The public surface described in REPS
section 4 is the spec the `0.3.0` implementation will conform to.
Consumers waiting on a functional release should pin against
`0.3.0` once it ships.

## Stability commitment

`0.2.0` is a minor release in the 0.x line. The pre-1.0 stability
convention for this crate is:

- **Minor (`0.x.0`)** - public surface may add and rearrange.
- **Patch (`0.x.y`)** - bug fixes only; no surface change.

API freeze begins at `1.0.0-rc.1`. Until then the design lock is
informational, not contractual.
